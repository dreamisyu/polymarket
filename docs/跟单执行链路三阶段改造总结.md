# 跟单执行链路三阶段改造总结

## 背景

本次改造围绕三个目标展开：

1. 降低真实资金执行风险，避免“下单成功但链上失败”被误记为已完成。
2. 提高源账户快照与执行决策的数据准确性，减少漏单、误判和错误比例缩放。
3. 压缩端到端延迟，并让模拟执行与实盘执行尽量共用同一套定价和风控逻辑。

---

## 阶段一：实盘风险收敛

### 目标

- 将真实下单从“提交成功即完成”改为“提交成功后等待最终确认”。
- 在下单前增加余额、授权、持仓和滑点保护，降低意外亏损概率。
- 增加卡死恢复与幂等保护，避免 `PROCESSING` 或 `SUBMITTED` 记录永久悬挂。

### 主要改动

- 为真实执行链路引入 `SUBMITTED` 状态，保存 `orderId`、`transactionHashes`、提交时间与错误信息。
- 加入 `PROCESSING_LEASE_MS` 租约回收，超时后允许重新认领未真正提交的记录。
- 下单前增加 CLOB 授权额度和可用余额检查，不再只依赖链上 `balanceOf`。
- 固定 `0.05` 美元滑点保护改为基于 `MAX_SLIPPAGE_BPS` 的相对滑点保护。
- 所有关键持仓查询统一带 `sizeThreshold=0`，避免小仓位被接口默认阈值吞掉。
- 最终确认链路已补充为官方 `User Channel WebSocket`：
    - 订阅 `wss://ws-subscriptions-clob.polymarket.com/ws/user`
    - 订阅消息直接带 `apiKey / secret / passphrase`
    - 监听 `MATCHED -> MINED -> CONFIRMED`、`RETRYING`、`FAILED`
    - 若运行时无 WebSocket 或缺失 `orderId`，仍保留链上回执轮询作为兜底

### 影响文件

- `src/services/tradeExecutor.ts`
- `src/services/clobUserStream.ts`
- `src/utils/postOrder.ts`
- `src/utils/getTradingGuardState.ts`
- `src/utils/confirmTransactionHashes.ts`
- `src/interfaces/User.ts`
- `src/models/userHistory.ts`

## 阶段二：监控与快照准确性

### 目标

- 把监视器从全量轮询改为增量同步，减少错位快照和重复扫描。
- 区分“未知”与“真实为 0”，避免错误比例缩放和误卖满仓。
- 将非交易类活动纳入持仓校准，降低源账户与本地仓位长期偏离。

### 主要改动

- 引入 `activityKey` 作为源活动幂等键，避免单纯依赖 `transactionHash` 造成多条 activity 覆盖。
- 新增 `executionIntent`，将活动区分为：
    - `EXECUTE`：真实需要自动执行的 `TRADE`
    - `SYNC_ONLY`：只用于仓位校准的 `MERGE / REDEEM`
- 引入 `UserActivitySyncState`，保存 `lastSyncedTimestamp` 和 `lastSyncedActivityKey`。
- 监视器改为按时间窗口增量拉取，并支持分页拉取官方 activity 接口。
- 所有快照缺失不再写入 `0`，而是写入：
    - `snapshotStatus = COMPLETE / PARTIAL / STALE`
    - `sourceSnapshotReason`
    - `sourceSnapshotCapturedAt`
- 执行器和模拟器只消费 `COMPLETE` 快照，避免对不完整快照做真实资金决策。

### 影响文件

- `src/services/tradeMonitor.ts`
- `src/utils/buildTradeSnapshots.ts`
- `src/utils/buildActivityKey.ts`
- `src/interfaces/User.ts`
- `src/models/userHistory.ts`

## 阶段三：延迟优化与模拟一致性

### 目标

- 减少每笔执行对 REST order book 的串行依赖。
- 让模拟器与实盘复用同一套盘口消费、分 chunk 执行和滑点判断逻辑。
- 降低监控和执行侧的重复 IO 与重复计算。

### 主要改动

- 引入市场缓存流，优先从官方 `Market Channel` WebSocket 维护盘口快照，HTTP 仅作兜底。
- 抽取共享执行规划层：
    - 统一计算买卖目标规模
    - 统一滑点保护
    - 统一 chunk 执行与盘口流动性消费
- `postOrder` 改为基于市场缓存和共享 planner 下单。
- `paperTradeExecutor` 不再直接按源成交价记账，改为按本地市场快照模拟吃单。
- 模拟链路改为按 `activityKey` 去重与落库，修复“同一链上交易多个 activity 被吞掉”的问题。
- 活动集合与 trace 执行集合增加关键索引，减少轮询与查询放大。

### 影响文件

- `src/services/clobMarketStream.ts`
- `src/services/createExecutor.ts`
- `src/services/paperTradeExecutor.ts`
- `src/utils/executionPlanning.ts`
- `src/utils/postOrder.ts`
- `src/models/traceHistory.ts`
- `src/interfaces/Trace.ts`

## 验证情况

- 已通过相关 TypeScript 文件的 ESLint。
- 当前环境为 Node `v25.3.0`，全量 `tsc --noEmit` 与定向 `tsc` 仍然会 OOM，暂时无法提供完整类型编译通过背书。
- 本次提交没有包含联网联调，也没有运行真实下单验证；仍需结合本地 `.env`、MongoDB 和目标钱包做联调。

## 已知边界与上线前注意事项

- `User Channel` 依赖服务端 WebSocket 运行时；如果运行环境不支持 WebSocket，会自动退回链上回执轮询兜底。
- 若历史 trace 执行集合上仍存在旧的 `sourceTransactionHash_1` 唯一索引，需要在部署前执行一次索引迁移或 `syncIndexes()`，否则同一 `transactionHash` 下的多条 activity 仍可能互相冲突。
- `SUBMITTED` 记录的最终确认现在优先依赖 `orderId` 监听；如果第三方接口返回异常缺失 `orderId`，会自动回退到 `transactionHash` 回执确认。
- 建议先做 24 小时 shadow 观察，再做小额 canary 实盘，最后再放开单笔和单市场风控上限。
