# 代码流程与架构

## 运行总览

当前运行时已经拆成两条长期任务：

1. `tradeMonitor` 负责采集和标准化源账户活动
2. 执行器负责把活动转换成可执行批次，并落到 `trace` 或 `live`
3. 结算 worker 负责已 resolved 市场的自动结算、回收和扫尾

共享主链路如下：

```text
Polymarket activity / positions
    -> tradeMonitor
    -> UserActivity
    -> CopyExecutionBatch
    -> trace 或 live 执行网关
    -> 审计 / 持仓 / 确认
    -> settlement worker
    -> 回收 / 结算
```

## 流程说明

### 1. 启动阶段

`src/index.ts` 只做三件事：

1. 连接 MongoDB
2. 通过 `createExecutor.ts` 按 `EXECUTION_MODE` 选择 `paperTradeExecutor` 或 `tradeExecutor`
3. 通过 `createSettlementWorker.ts` 装配对应模式的结算 worker
4. 并发启动 `tradeMonitor`、执行器与结算 worker

### 2. 监控阶段

`src/services/tradeMonitor.ts` 负责把源账户活动转换成可消费流水：

1. 读取 `UserActivitySyncState`
2. 拉取 Polymarket `activity` 接口
3. 过滤并标准化 `TRADE / MERGE / REDEEM`
4. 生成 `activityKey`
5. 结合源账户余额和仓位生成成交后快照
6. 写入 `user_activities_*`

这里的设计重点是把“源事件事实”与“执行结果”分离。`UserActivity` 永远是源账户原始事件账本，不直接等同于真实下单记录。

### 3. 执行编排阶段

`tradeExecutor.ts` 和 `paperTradeExecutor.ts` 共享同一套编排语义：

1. 从 `UserActivity` 读取 `PENDING` 或可回收的待处理源交易
2. 监视器先把“同市场、同方向、相邻仓位”的源买单合并成单条 `UserActivity`
3. 执行器对单条买单按“源账户买入前现金 / 代理账户当前现金”做比例定额
4. 若单条买单 `< 1 USDC`，只做补齐或直接跳过，不再维护执行层买单缓冲区
5. 对可执行批次调用 `executionPlanning.ts` 生成盘口分片计划
6. 执行完成后，把状态回写到批次和关联源交易

这层的关键目标是把“源交易事实”和“执行批次”解耦，同时把多笔相邻小买单的合并职责前移到监视器，减轻执行器状态机负担。

### 4. trace 分支

`src/services/paperTradeExecutor.ts` 负责模拟执行：

- 读取公开盘口
- 在本地消费流动性并计算成交均价
- 更新 `trace_positions_*` 与 `trace_portfolios_*`
- 以批次维度记录 `trace_executions_*`

trace 不走真实下单，也不走链上确认；它的职责是验证策略行为和组合表现。

### 5. live 分支

`src/services/tradeExecutor.ts` 负责真实执行：

- 使用 `postOrder.ts` 调 Polymarket 下单
- 优先等待 `ClobUserStream` 的 User Channel 生命周期回报
- User Channel 不可用或超时时，回退到 `confirmTransactionHashes.ts` 做链上确认
- 空闲轮次由 `liveSettlementReclaimer.ts` 自动回收 `redeemable` 仓位

live 的重点不是只下单成功，而是把“提交、撮合、上链、确认、回收”串成可补偿的完整链路。

## 关键状态流

### 源交易状态

```text
PENDING
  -> BATCHED
  -> SUBMITTED
  -> CONFIRMED / SKIPPED / FAILED
```

### 执行批次状态

```text
READY
  -> PROCESSING
  -> SUBMITTED       live
  -> CONFIRMED / SKIPPED / FAILED
```

trace 批次通常会直接从 `PROCESSING` 落到终态；live 批次会多一个 `SUBMITTED` 和确认补偿阶段。

## 设计约束

- `tradeMonitor` 只写源事件，不在监控侧直接做真实执行
- `CopyExecutionBatch` 才是执行与确认的最小单元
- 相邻小买单的合并发生在 `tradeMonitor`，执行器只处理合并后的单条活动
- `trace` 与 `live` 共用直接入批语义，但落到不同集合与不同执行网关
- `min_order_size` 仍沿用平台返回值与现有判断逻辑，本次架构不改这个契约
