# 核心跟单流程与推荐配置

## 入口总览

`src/index.ts` 现在只保留 3 个职责：

1. 连接 MongoDB
2. 根据 `EXECUTION_MODE` 创建对应执行器
3. 并发启动监视器与执行器，并在异常退出时输出统一错误日志

因此运行时结构可以直接理解为：

```text
index.ts
├─ tradeMonitor            持续抓取源账户活动并写入 MongoDB
└─ executor
   ├─ paperTradeExecutor   trace 模式，本地模拟记账
   └─ tradeExecutor        live 模式，真实下单并补偿确认
```

## 监视器逻辑

`src/services/tradeMonitor.ts` 负责把源账户活动整理成可消费的标准流水。

执行顺序如下：

1. 读取 `UserActivitySyncState`，确定本轮抓取窗口
2. 调用 Polymarket `activity` 接口，按时间窗口分页拉取活动
3. 过滤非 `TRADE / MERGE / REDEEM` 活动，并统一时间戳、类型与 `activityKey`
4. 读取源账户当前持仓与余额，生成每笔活动的成交后快照
5. 将活动写入 `user_activities_*` 集合，同时补齐执行意图与快照状态
6. 更新同步游标，供下一轮继续增量抓取

关键产物：

- `executionIntent=EXECUTE`：只给 `TRADE`，供执行器消费
- `executionIntent=SYNC_ONLY`：给 `MERGE / REDEEM`，只用于持仓校准
- `snapshotStatus=COMPLETE / PARTIAL / STALE`：决定后续是否允许执行

## 模拟执行器逻辑

`src/services/paperTradeExecutor.ts` 只在 `EXECUTION_MODE=trace` 下运行。

执行顺序如下：

1. 从 `user_activities_*` 中读取可执行且未记账的源交易
2. 用 `trace_executions_*` 做幂等占位，避免重复模拟
3. 根据源账户成交后持仓与本地模拟持仓，计算执行条件：`buy / sell / merge`
4. 从市场快照读取盘口，按与实盘一致的分片规划逻辑撮合
5. 更新 `trace_positions_*` 与 `trace_portfolios_*`
6. 记录 `FILLED / SKIPPED` 结果到 `trace_executions_*`
7. 空闲时刷新持仓市值，并对 `redeemable` 仓位自动结算

这条链路的作用是先验证两个核心问题：

- 跟单比例是否符合预期
- 当前风控参数下，组合权益曲线是否合理

## 实盘执行器逻辑

`src/services/tradeExecutor.ts` 只在 `EXECUTION_MODE=live` 下运行。

执行顺序如下：

1. 从 `user_activities_*` 读取 `PENDING` 或超时回收后的待执行交易
2. 领取处理租约，防止多实例重复执行同一笔交易
3. 拉取代理钱包持仓与真实交易风控状态
4. 校验余额、授权、未完成挂单、源账户快照是否完整
5. 计算执行条件并调用 `postOrder` 分片下单
6. 将结果写回源交易记录：`SUBMITTED / SKIPPED / FAILED`
7. 优先使用 User Channel 跟踪订单生命周期；不可用时退回链上确认
8. 最终落库为 `CONFIRMED / FAILED`，未确认完成的交易会回到待补偿状态

实盘链路有两个关键保护：

- `PROCESSING_LEASE_MS`：防止 worker 异常退出后记录永久卡死
- `RETRY_LIMIT`：限制重复下单次数，避免异常行情下无限重试

## 日志策略

本次整理后，日志只保留以下节点：

- 应用启动
- 监视器同步摘要
- 执行器发现待处理交易
- 单笔交易的执行、提交、确认、跳过、失败、重试
- WebSocket / HTTP / 余额读取等基础设施异常

已经去掉的高噪音内容：

- 每次 HTTP 请求成功日志
- 每笔订单完整参数与原始响应
- 整个交易对象的原样打印
- 装饰性表情与无业务意义文本

## 推荐配置

下面给出两套建议基线。原则是先 `trace`，确认行为稳定后再切 `live`。

TRACE_ID=paper

FETCH_INTERVAL=1

ACTIVITY_SYNC_OVERLAP_MS=30000

## 参数建议

建议优先关注这几个可调项：

- `MAX_ORDER_USDC`
  首次实盘建议从 `25-50` 开始，先验证资金比例与成交行为，再逐步放开。
- `MAX_SLIPPAGE_BPS`
  默认 `300` 比较均衡；如果你主要跟高流动性市场，可下调到 `150-200`。
- `FETCH_INTERVAL`
  `1` 秒延迟最低；如果你更在意接口压力而不是极限跟单速度，可调到 `2-3` 秒。
- `SNAPSHOT_STALE_AFTER_MS`
  超过这个窗口的源账户快照会被标记为陈旧并跳过执行，建议不要高于 `60000`。
- `PROCESSING_LEASE_MS`
  单实例部署建议维持 `30000`；如果链路常出现长时间阻塞，再按实际耗时放大。

## 落地顺序

推荐按下面顺序使用：

1. 用 `.env.trace` 跑一段时间，观察 `trace_portfolios_*` 和执行汇总
2. 调整 `MAX_ORDER_USDC`、`MAX_SLIPPAGE_BPS`
3. 确认模拟结果稳定后，再切换 `.env.live`
4. 实盘初期保持较小 `MAX_ORDER_USDC`，先验证确认链路与日志输出
