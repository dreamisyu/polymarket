# 分析脚本说明

`scripts/` 目录现在包含一组面向“定位系统改进点”的脚本，目标不是替代线上监控，而是让后续复盘能快速回答这几个问题：

- 瓶颈主要在监视器、执行器还是结算回收。
- 小额碎单、滑点、快照缺失、结算积压分别占多大比例。
- 目标账户本身的交易画像是否天然不适合当前复制策略。
- 运行日志里反复出现的热点异常是否已经和 DB 里的结果对应上。

## 脚本清单

### `trace-settlement-audit.mjs`

用途：

- 复核 trace 数据集与 Polymarket 官方市场状态的结算一致性。

典型命令：

```bash
npm run report:trace-audit -- --trace-id default --json
```

### `execution-summary.mjs`

用途：

- 查看 live / trace 的执行漏斗、批次状态和 Polymarket 持仓汇总。

典型命令：

```bash
npm run report:summary -- --mode live --json
```

### `db-improvement-report.mjs`

用途：

- 从 MongoDB 直接汇总 `user_activities`、`copy_execution_batches`、`trace_executions`、`trace_settlement_tasks`。
- 输出当前系统最值得优先优化的方向，例如小额买单占比、快照质量、结算积压、legacy buffer 残留。

典型命令：

```bash
npm run report:db-improvement -- --mode trace --trace-id default --hours 12
```

适用场景：

- 改完执行链路后，先看 DB 漏斗是否真的收敛。
- 判断问题主要出在入口活动、执行批次，还是结算回补。

### `log-improvement-report.mjs`

用途：

- 从运行日志中抽取热点问题分类、热点 scope、热点 condition/asset。
- 适合排查 403/429、orderbook 404、执行异常、快照缺失、结算重试等问题。

典型命令：

```bash
npm run report:log-improvement -- --file logs/bot.log
cat logs/bot.log | npm run report:log-improvement -- --json
```

适用场景：

- DB 数据已经说明“有问题”，但还不清楚是哪条运行链路在反复报错。
- 需要把日志热点和 DB 结果按 condition 对齐。

### `target-wallet-profile.mjs`

用途：

- 基于目标账户活动与官方持仓接口，分析目标账户的交易画像。
- 重点回答：碎单多不多、MERGE/REDEEM 占比高不高、当前相邻 BUY 还有多少可继续合并空间。

典型命令：

```bash
npm run report:target-profile -- --hours 24 --merge-window-ms 15000
```

适用场景：

- 需要判断“问题出在系统”，还是“目标账户交易风格本来就不适合当前复制策略”。
- 需要评估是否继续扩大监视器相邻合并窗口。

## 推荐排查顺序

### 1. 先看 DB 漏斗

```bash
npm run report:db-improvement -- --mode trace --trace-id default
```

优先回答：

- 跳过最多的是哪类原因。
- 小额买单是否仍是主损耗点。
- 结算任务是否有积压。

### 2. 再看运行日志

```bash
npm run report:log-improvement -- --file logs/bot.log
```

优先回答：

- 403/429 是否还在出现。
- resolved 市场是否仍访问 orderbook。
- 执行异常是否主要集中在某些 condition。

### 3. 最后看目标账户画像

```bash
npm run report:target-profile -- --hours 24
```

优先回答：

- 目标账户是否天然偏碎单。
- MERGE/REDEEM 是否足够频繁，值得继续优化 condition 净额模型。
- 当前监视器窗口下，仍有多少相邻 BUY 可以继续被聚合。

## 注意事项

- 脚本默认读取项目根目录或当前工作目录的 `.env`。
- 涉及 Polymarket 官方接口的脚本可能受网络和限流影响，建议优先在本地重跑并结合缓存观察。
- 这些脚本面向“诊断与复盘”，不应直接接入自动化写库流程。
